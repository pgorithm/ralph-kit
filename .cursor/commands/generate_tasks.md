# generate_tasks — генерация очереди из PRD и currentProblems

Ты — инициализирующий агент для рабочего батча доработок. Сгенерируй **`docs/tasks/tasks.json`** с **новыми** задачами на основе PRD и списка проблем так, чтобы очередь можно было исполнять инкрементально (одна задача за сессию или K параллельных воркеров).

## Контекст процесса

1. `docs/tasks/currentProblems.md` хранит сырьё: баги, идеи, полные ТЗ.
2. `update_prd` переносит это сырьё в `{{PRD_PATH}}` как структурированные требования.
3. Этот промпт превращает требования в атомарные TASK-задачи, учитывая уже выполненное в `docs/tasks/progress.md`.

## Входные данные

1. **`{{PRD_PATH}}`** — раздел **`{{CURRENT_WORK_SECTION}}`** (текущие доработки) и смежные разделы.
2. **`docs/tasks/progress.md`** — источник последнего TASK-ID и уже закрытых работ.
3. **`docs/tasks/currentProblems.md`** — полный контекст проблем/ожиданий.
4. **`docs/new-agents.md`** — правила Build & Test и gotchas.

## Выходной файл

- **Путь:** `docs/tasks/tasks.json`
- **Формат:** `docs/tasks/tasks.template.json`
- **Содержимое:** только новые задачи, не дублирующие закрытое в progress
- **Нумерация:** продолжай после последнего TASK в progress
- **Стартовый статус:** `pending`

## Требования к задачам

### 1) Обязательная структура

Используй JSON, не Markdown. Копируй блок **`agent_instructions`** из `docs/tasks/tasks.template.json` **целиком** (не сокращай). Каждая задача — все поля из шаблона (`assignee`, `claimed_at`, `lease_until`, `review_required`, `status_reason`, `artifacts`, `test_verdict`, …).

### 2) Декомпозиция

- Одна задача = один завершимый инкремент (ориентир: до 30 минут активной реализации).
- Минимизируй зависимости; критический путь выстраивай явно.
- Формулируй проверяемые `acceptance_criteria` и практичные `test_steps`.
- Приоритет по умолчанию: инфраструктура/безопасность/контракты -> core-функции -> UI/полировка.

### 3) Категоризация

Выбирай по смыслу: `infrastructure`, `functional`, `integration`, `ui`, `security`.

### 4) Задачи без изменения кода

Если задача касается только документации/процесса/очереди:

- Явно пометь в `description` или первом AC: **«без изменений кода»**.
- В `test_steps` опиши ручную/документальную проверку.
- Не требуй полного автотестового прогона для такой задачи.

### 5) Release metadata

При создании нового `docs/tasks/tasks.json` добавь верхнеуровневый блок `release` по форме `docs/tasks/tasks.template.json`. Семантика — `docs/project/product-versioning.md`.

| Поле | Как заполнить |
|------|----------------|
| `kind` | `patch` \| `minor` \| `major` — **по всему батчу**, не по одной TASK |
| `baseline_version` | опционально, из `{{VERSION_FILE}}` до батча |
| `notes` | краткий контекст для closure |

Классификация `kind` (доминирующее содержимое батча):

- **`patch`** — техдолг, security-fixes, баги, infra-hardening, docs/process без заметных пользовательских фич.
- **`minor`** — новые пользовательские сценарии, заметные улучшения UI/API/интеграций.
- **`major`** — только при **явном** критерии в PRD/запросе (breaking API, prod-монетизация `0→1`, смена архитектурной модели).

Смешанный батч — одна цифра на релиз: побеждает **старшая** (`functional` + `security` → `minor`, не `patch`).

Блок `release` — подсказка для closure; **не** триггер bump в середине батча.

## Проверка перед сохранением

1. Нет задач, семантически дублирующих уже закрытые в `progress.md`.
2. Нумерация непрерывная.
3. В зависимостях нет циклов.
4. Не осталось плейсхолдеров вида `TODO`/`...`.

## После генерации

В отчёте укажи:

- сколько задач создано;
- диапазон ID;
- есть ли блок `release`;
- рекомендованный следующий шаг: `run_ralph` (solo) или `run_ralph_orchestrator` (parallel).
