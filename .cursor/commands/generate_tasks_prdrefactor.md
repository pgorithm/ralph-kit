# generate_tasks_prdrefactor — очередь рефакторинга PRD

Ты — инициализирующий агент. По **полному** `{{PRD_PATH}}` и **фактической реализации** сформируй очередь задач на **редактирование PRD** (без изменения production-кода).

## Цель

1. Привести PRD к **одной непротиворечивой** картине (все разделы, не только «текущие доработки»).
2. Перенести накопившиеся пункты из раздела текущих доработок (например **§3.5**) в основные разделы; сократить §3.5 до отсылок.
3. Устранить расхождения «PRD ↔ код ↔ progress»; приоритет: **реализация** → **progress** → более новый текст PRD.

## Входные данные

1. **`{{PRD_PATH}}`** — целиком.
2. **`docs/tasks/progress.md`** — что уже сделано.
3. **Код и конфигурация** — выборочно по темам расхождений.
4. **Тесты** — вторичный источник ожидаемого поведения.

## Выходной файл

- **Путь:** `docs/tasks/prd_refactor_tasks.json`
- **Шаблон:** `docs/tasks/prd_refactor_tasks.template.json`
- **ID задач:** `PRD-REFACTOR-001`, `PRD-REFACTOR-002`, … (отдельно от `TASK-xxx`)
- **Статус новых задач:** `pending`

## Формат задачи

Каждая задача по `prd_refactor_tasks.template.json`, включая поля:

- `target_section` — куда переносить канон (например `3.1`, `3.2`)
- `sections_to_merge` — список подпунктов §3.5.X для переноса; `[]` если задача только вычитка без merge

## Декомпозиция

- Одна задача = один логический кусок PRD, укладывается в одну сессию агента.
- Явные `dependencies` между PRD-REFACTOR при порядке переноса.
- `test_steps` — ручная проверка PRD (ссылки, оглавление, отсутствие дублей).

## agent_instructions

- при оркестраторном режиме — atomic claim по `run_ralph_orchestrator.md`;
- в solo — `run_ralph_prdrefactor.md`.
